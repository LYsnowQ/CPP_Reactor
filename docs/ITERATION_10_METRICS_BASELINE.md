# Iteration 10 - 可观测性与基线压测（方案 A）

## 本轮目标
- 给 Demo 项目补齐最小工业化可观测能力：指标采集 + 窗口日志输出。
- 提供可执行压测脚本与基线报告模板，形成“改动 -> 压测 -> 评估”的闭环。
- 用术语解释帮助首次接触压测的同学快速读懂日志和报告。

## 改动理由（为什么要这样改）
- 改动前问题：
  - 项目能跑，但无法量化“跑得怎么样”。
  - 缺少统一指标口径，优化前后无法客观对比。
- 改动后解决：
  - 增加统一指标聚合器，周期打印关键性能指标。
  - 增加基线脚本与报告模板，支持重复测量与版本对比。

## 技术点与代码位置

### 1) 统一指标聚合器（线程安全）
- 技术点：
  - 新增 `Metrics` 单例，使用原子计数聚合跨线程指标。
  - 指标覆盖：连接、请求、响应状态分布、读写字节、请求时延。
- 代码位置：
  - `Metrics.hpp`
  - `Metrics.cpp`

### 2) 连接与请求埋点
- 技术点：
  - 在 `TcpConnection` 记录读写字节、请求完成计数、响应状态码与请求时延。
  - 时延口径：从收到请求首包到生成响应头这一刻的耗时（微秒）。
- 代码位置：
  - `TcpConnection.hpp`
  - `TcpConnection.cpp`

### 3) 周期窗口日志（5秒）
- 技术点：
  - 在 `TcpServer` 每 5 秒计算增量窗口，输出 QPS、连接、响应分布、吞吐、平均时延。
  - 使用“窗口增量”而非总量，便于观测实时趋势。
- 代码位置：
  - `TcpServer.hpp`
  - `TcpServer.cpp`

### 4) 压测脚本与报告模板
- 技术点：
  - 提供 `scripts/bench_http.sh`，优先 `wrk`，其次 `ab`，最后 `curl` 兜底。
  - 提供 `docs/PERF_BASELINE_TEMPLATE.md`，统一记录格式。
- 代码位置：
  - `scripts/bench_http.sh`
  - `docs/PERF_BASELINE_TEMPLATE.md`

## 日志术语解释（你看日志时重点关注）
- `qps`：每秒请求数，衡量吞吐能力。
- `active_conn`：当前活跃连接数，衡量连接压力。
- `accept_ok / accept_fail`：连接接入成功/失败数，衡量接入稳定性。
- `resp_2xx / 4xx / 5xx`：响应分类计数，衡量成功率和错误类型。
- `read_Bps / write_Bps`：网络读写速率，衡量链路负载。
- `avg_latency_us`：窗口平均请求时延（微秒），衡量响应速度。
- `max_latency_us`：全局观测最大时延，衡量尾延迟风险。

## 怎么用（建议流程）
1. 启动服务并观察 5 秒窗口日志。
2. 运行 `./scripts/bench_http.sh http://127.0.0.1:8080/ 20s 200 4`。
3. 将输出结果按模板填写到 `docs/PERF_BASELINE_TEMPLATE.md`。
4. 后续每次优化后复测，做横向对比。

## 相比上一版本解决的问题
- 相比 Iteration 09，本轮解决了“功能可运行但缺乏性能量化手段”的问题。
- 建立了最小可观测闭环，为收尾阶段的性能评估和学习复盘提供依据。

## 已知边界
- 当前未接入 Prometheus 导出，指标以日志为主。
- 时延统计是基础口径（均值+最大值），未包含 P95/P99 分位。
