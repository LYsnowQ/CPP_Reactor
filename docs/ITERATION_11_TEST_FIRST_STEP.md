# Iteration 11 - 测试体系第一步（冒烟 + 协议用例 + 单次基线）

## 本轮目标
- 落地测试第一步：从“可运行”升级到“可重复验证”。
- 给后续优化提供统一测试入口，避免每次手工拼命令。

## 改动前问题
- 缺少固定测试入口，回归依赖人工操作，容易漏测。
- 缺少协议边界样本，难以稳定复现解析问题。
- 压测已有脚本但缺少标准执行流程，不利于做版本对比。

## 技术点与代码位置

### 1) 测试蓝图文档
- 技术点：
  - 新增测试目录说明与执行顺序，约束统一测试流程。
- 代码位置：
  - `tests/README.md`

### 2) 协议用例运行器（无第三方依赖）
- 技术点：
  - 使用 Python socket 直接发原始 HTTP 报文。
  - 覆盖四类样本：合法请求、非法请求、超大包、半包 body。
  - 输出 PASS/FAIL 与摘要，便于 CI 前置接入。
- 代码位置：
  - `tests/run_http_cases.py`

### 3) 冒烟脚本
- 技术点：
  - 用 curl 快速验证服务可达性与基础响应码。
  - 作为每次改动后的第一道低成本检查。
- 代码位置：
  - `scripts/test_smoke.sh`

### 4) 单次基线脚本
- 技术点：
  - 封装单次压测入口，调用现有 `bench_http.sh`。
  - 输出参数和提示，便于将结果回填到基线模板。
- 代码位置：
  - `scripts/test_baseline_once.sh`

## 术语解释（本轮涉及）
- 冒烟测试（Smoke Test）：
  - 用最小成本验证“服务是否活着、主路径是否可达”。
- 协议用例（Protocol Cases）：
  - 针对 HTTP 报文结构和边界条件的功能验证。
- 基线（Baseline）：
  - 某一版本在固定参数下的参考性能数据，用于后续对比优化效果。

## 相比上一版本解决的问题
- 相比 Iteration 10，本轮把“可观测”推进到“可测试流程化”。
- 建立了第一批可执行测试入口，后续可直接扩展为自动化回归。

## 下一步建议
- 增加 `tests/http_cases/*.txt` 的外部样本文件，避免样本硬编码在脚本中。
- 增加 30 分钟稳定性测试脚本，观察错误率和指标漂移。
